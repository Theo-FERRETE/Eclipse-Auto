01
Vue d'ensemble

Eclipse Auto est une application web full-stack simulant une concession automobile haut de gamme. Les visiteurs peuvent consulter le catalogue, les clients connectés peuvent réserver un véhicule, et les administrateurs disposent d'un back-office complet.

Fonctionnalités par rôle

RôleFonctionnalités
VisiteurCatalogue avec filtres et recherche, fiche véhicule, formulaire de contact
ClientInscription, connexion, réservation de véhicule, suivi et annulation, modification du profil
AdminTableau de bord statistiques, CRUD véhicules, gestion des réservations (confirmer/annuler), liste clients

Design system maison

02
Structure Frontend

Le frontend est une Single Page Application React construite avec Vite. Chaque page et composant est isolé dans son propre dossier avec son fichier JSX et son CSS dédié.

Pages

client/src/pages/
Home/
Page d'accueil — hero, véhicules mis en avant, sections marketing
Catalogue/
Liste paginée des véhicules avec filtres (marque, carburant, prix, statut), recherche texte, tri et mises à jour en temps réel via Supabase Realtime
VehicleDetail/
Fiche détaillée d'un véhicule — galerie d'images, spécifications, prix, statut, bouton réserver (si disponible)
Reservation/
Formulaire de réservation — message personnalisé, date de RDV souhaitée. Accessible uniquement aux clients connectés
Dashboard/
Espace client — deux vues : liste des réservations avec possibilité d'annulation, et modification du profil (prénom, nom, téléphone, mot de passe)
Login/ & Register/
Connexion et inscription via Supabase Auth. Création automatique du profil à l'inscription
Contact/
Formulaire de contact envoyé au serveur Express via POST /api/contact
MentionsLegales/
Page des mentions légales — informations sur l'éditeur, l'hébergeur et la politique de confidentialité
NotFound/
Page 404 — redirige vers l'accueil ou le catalogue

Pages Admin

client/src/pages/admin/
AdminDashboard/
Statistiques globales — total véhicules, réservations par statut, nombre de clients, CA estimé
AdminVehicles/
CRUD complet — ajouter, modifier et supprimer des véhicules via un panneau latéral. Changement de statut rapide. Pagination et recherche
AdminReservations/
Gestion des réservations — confirmer ou annuler. Le statut du véhicule est mis à jour automatiquement par un trigger PostgreSQL
AdminUsers/
Liste des clients inscrits avec suppression via modale de confirmation

Composants

Les composants sont des blocs d'interface réutilisables, indépendants des pages. Chaque composant expose des props pour le personnaliser.

client/src/components/
Navbar/
Navigation principale responsive. Affiche le prénom de l'utilisateur connecté. Le lien "Réservations" est masqué si non connecté. Menu burger sur mobile avec aria-label et aria-expanded
Footer/
Pied de page avec liens, mentions légales et informations de contact
VehicleCard/
Carte véhicule affichée dans le catalogue — image principale, marque, modèle, année, carburant, prix et badge de disponibilité. Lien vers la fiche détail
Filters/
Panneau de filtres du catalogue. Marques générées dynamiquement depuis les véhicules chargés. Filtres : marque, carburant, transmission, prix max (slider), année, statut
Pagination/
Pagination intelligente — affiche au maximum 7 boutons avec ellipsis (…) pour les longues séries. Boutons précédent/suivant avec aria-labels pour l'accessibilité
AdminSidebar/
Barre de navigation commune à toutes les pages admin. Lien actif détecté automatiquement via l'URL courante
ConfirmModal/
Modale de confirmation stylisée. Remplace les boîtes de dialogue natives du navigateur pour les suppressions d'éléments
ProtectedRoute/
Composant de protection des routes. Redirige vers /login si non connecté, vers l'accueil si la page requiert un rôle admin et que l'utilisateur ne l'a pas

client/src/lib/
AuthContext.jsx
Contexte React global d'authentification. Expose à tous les composants : l'utilisateur connecté, son profil, son rôle (isAdmin / isClient), et une fonction refreshProfile() pour recharger les données après modification

Initialisation du client Supabase avec la clé publique (anon key). Utilisé pour les requêtes depuis le navigateur

Fonctions d'authentification — connexion, inscription, déconnexion via Supabase Auth

Constantes centralisées partagées dans toute l'application — statuts des réservations (RESERVATION_STATUS), statuts des véhicules (VEHICLE_STATUS), types de carburant (FUEL_TYPES), transmissions (TRANSMISSIONS), et la fonction toSlug() pour générer les URLs des véhicules

Fichiers racine

client/src/
main.jsx
Point d'entrée React — monte l'application dans le DOM et enveloppe l'arborescence dans <AuthProvider> pour rendre le contexte d'authentification accessible globalement
App.jsx
Définition complète du routeur React Router v7 — déclare toutes les routes publiques, protégées (client) et admin. Utilise <ProtectedRoute> pour sécuriser les accès selon le rôle

04
Structure Backend

Le backend est un serveur Express.js 5 qui tourne sur le port 3001. Il gère les opérations sensibles qui ne doivent pas être faites depuis le navigateur — notamment les vérifications métier et l'accès à la base de données avec des droits élevés. Apache redirige automatiquement toutes les requêtes /api/* vers ce serveur.

Fichiers du serveur

server/
index.js
Point d'entrée du serveur — initialise Express, configure CORS, branche les routeurs sur leurs préfixes, gère les erreurs 404 et globales
supabase.js
Client Supabase avec la clé service_role — cette clé bypass toutes les restrictions RLS de la base de données. Utilisée uniquement côté serveur, jamais exposée au client
middleware/auth.js
Deux middlewares de protection des routes — requireAuth vérifie le token JWT dans l'en-tête Authorization, requireAdmin vérifie en plus que le rôle dans la table profiles est "admin"
routes/vehicles.js
Routes véhicules — lecture publique, création/modification/suppression réservées aux admins
routes/reservations.js
Routes réservations — création avec vérification de disponibilité en temps réel, annulation client avec contrôle de propriété, gestion admin
routes/admin.js
Routes d'administration — statistiques globales (8 requêtes parallèles), liste et suppression de clients
routes/contact.js
Réception du formulaire de contact et traitement du message

05
Flux d'authentification

L'authentification repose entièrement sur Supabase Auth (JWT). Le frontend génère et stocke le token ; le backend le valide sans jamais manipuler les mots de passe.

Inscription d'un client

1
L'utilisateur remplit le formulaire Register (prénom, nom, e-mail, mot de passe)
2
Supabase Auth crée le compte dans auth.users et retourne un JWT
3
Un enregistrement est inséré dans la table profiles avec le rôle "client"
4
AuthContext met à jour l'état global — l'utilisateur est immédiatement connecté

Connexion et accès aux routes protégées

1
L'utilisateur se connecte via la page Login — Supabase Auth retourne un JWT stocké en mémoire
2
Le composant ProtectedRoute lit le contexte — redirige vers /login si le token est absent
3
Chaque appel API inclut le JWT dans l'en-tête : Authorization: Bearer <token>
4
Le middleware requireAuth (server) valide le token via Supabase Admin
5
Pour les routes admin, requireAdmin vérifie en plus que profiles.role = "admin"

Schéma des rôles et permissions

RôleToken requisAccès autorisé
VisiteurNonCatalogue, fiches véhicules, contact, mentions légales
ClientJWT valideTout visiteur + créer/annuler réservations, modifier son profil
AdminJWT + rôle admin en BDDTout client + CRUD véhicules, gestion réservations, liste clients

Base de données

La base de données est hébergée sur Supabase (PostgreSQL managé). Les Row Level Security (RLS) sont activées sur toutes les tables : chaque utilisateur ne peut accéder qu'à ses propres données, même en appelant Supabase directement depuis le navigateur.

Table — profiles

ColonneTypeDescription
iduuidLié à auth.users de Supabase (créé à l'inscription)
first_nametextPrénom
last_nametextNom de famille
phonetextNuméro de téléphone (optionnel)
roletext"client" ou "admin"
created_attimestamptzDate d'inscription

Table — vehicles

ColonneTypeDescription
iduuidIdentifiant unique
brandtextMarque (ex : Ferrari)
modeltextModèle (ex : Roma Spider)
yearintegerAnnée de fabrication
pricenumericPrix en euros
fuel_typetextEssence / Diesel / Hybride / Électrique
transmissiontextAutomatique / Manuelle
mileageintegerKilométrage (0 = neuf)
powertextPuissance (ex : "620 CH")
statustext"available" / "reserved" / "sold" — mis à jour automatiquement par trigger

descriptiontextDescription libre (optionnel)

Table — reservations

ColonneTypeDescription

client_iduuidClé étrangère → profiles.id

rdv_datedateDate de rendez-vous souhaitée (optionnel)
messagetextMessage libre du client (optionnel)

Trigger PostgreSQL — Synchronisation automatique

Un trigger nommé sync_vehicle_status_on_reservation s'exécute automatiquement à chaque changement de statut d'une réservation, sans intervention manuelle.

1
L'admin confirme une réservation → son statut passe à confirmed
2
Le trigger détecte le changement dans la table reservations
3
Il met automatiquement vehicles.status = "reserved" pour le véhicule concerné
4
Si la réservation est annulée, il remet le statut du véhicule à "available"
5
Supabase Realtime propage la mise à jour en direct dans le catalogue pour tous les visiteurs
07
Déploiement

Architecture de production

1
Navigateur — envoie une requête HTTPS vers eclipse-auto.theo-ferrete.fr
2
Apache (port 443) — termine le SSL Let's Encrypt et sert les fichiers statiques React depuis client/dist
3
Pour les URLs /api/*, Apache fait un reverse proxy vers localhost:3001
4
Express (port 3001) géré par PM2 traite la requête et interroge Supabase
5
Supabase Cloud — base de données PostgreSQL hébergée externement
