Vue d'ensemble

Eclipse Auto est une application web full-stack simulant une concession automobile haut de gamme. Les visiteurs peuvent consulter le catalogue, les clients connectés peuvent réserver un véhicule, et les administrateurs disposent d'un back-office complet.

06
Structure Backend

Le backend est un serveur Express.js 5 qui tourne sur le port 3001. Il gère les opérations sensibles qui ne doivent pas être faites depuis le navigateur — notamment les vérifications métier et l'accès à la base de données avec des droits élevés. Apache redirige automatiquement toutes les requêtes /api/* vers ce serveur.

Fichiers du serveur

server/
index.js
Point d'entrée du serveur — initialise Express, configure CORS, branche les routeurs sur leurs préfixes, gère les erreurs 404 et globales
supabase.js
Client Supabase avec la clé service_role — cette clé bypass toutes les restrictions RLS de la base de données. Utilisée uniquement côté serveur, jamais exposée au client
middleware/auth.js
Deux middlewares de protection des routes — requireAuth vérifie le token JWT dans l'en-tête Authorization, requireAdmin vérifie en plus que le rôle dans la table profiles est "admin"
routes/vehicles.js
Routes véhicules — lecture publique, création/modification/suppression réservées aux admins
routes/reservations.js
Routes réservations — création avec vérification de disponibilité en temps réel, annulation client avec contrôle de propriété, gestion admin
routes/admin.js
Routes d'administration — statistiques globales (8 requêtes parallèles), liste et suppression de clients
routes/contact.js
Réception du formulaire de contact et traitement du message

Flux d'authentification

L'authentification repose entièrement sur Supabase Auth (JWT). Le frontend génère et stocke le token ; le backend le valide sans jamais manipuler les mots de passe.

Inscription d'un client

1
L'utilisateur remplit le formulaire Register (prénom, nom, e-mail, mot de passe)
2
Supabase Auth crée le compte dans auth.users et retourne un JWT
3
Un enregistrement est inséré dans la table profiles avec le rôle "client"
4
AuthContext met à jour l'état global — l'utilisateur est immédiatement connecté

Connexion et accès aux routes protégées

1
L'utilisateur se connecte via la page Login — Supabase Auth retourne un JWT stocké en mémoire
2
Le composant ProtectedRoute lit le contexte — redirige vers /login si le token est absent
3
Chaque appel API inclut le JWT dans l'en-tête : Authorization: Bearer <token>
4
Le middleware requireAuth (server) valide le token via Supabase Admin
5
Pour les routes admin, requireAdmin vérifie en plus que profiles.role = "admin"

Schéma des rôles et permissions

RôleToken requisAccès autorisé
VisiteurNonCatalogue, fiches véhicules, contact, mentions légales
ClientJWT valideTout visiteur + créer/annuler réservations, modifier son profil
AdminJWT + rôle admin en BDDTout client + CRUD véhicules, gestion réservations, liste clients

Trigger PostgreSQL — Synchronisation automatique

Un trigger nommé sync_vehicle_status_on_reservation s'exécute automatiquement à chaque changement de statut d'une réservation, sans intervention manuelle.

1
L'admin confirme une réservation → son statut passe à confirmed
2
Le trigger détecte le changement dans la table reservations
3
Il met automatiquement vehicles.status = "reserved" pour le véhicule concerné
4
Si la réservation est annulée, il remet le statut du véhicule à "available"
5
Supabase Realtime propage la mise à jour en direct dans le catalogue pour tous les visiteurs

Optimisation des images & Performance

L'application a subi un audit de performance complet en mai 2026. Les images représentaient 95% du poids téléchargé — elles ont été réduites de 2 à 6 MB chacune à 50-350 KB via conversion WebP.

Problème initial

ParamètreAvantAprèsGain
Audi RS3.jpg6.2 MB211 KB−97%
Mercedes AMG.jpg5.9 MB88 KB−98%
Format demandé (cartes)800px560px30% moins de requête
Format demandé (featured)600px420px30% moins de requête
Loading prioritaireslazy + aucune prioritéeager + fetchPriority=highImages visibles instantanément
Impact utilisateur : le site passe d'un chargement des images lent à des images visibles immédiatement au démarrage, sans flash blanc.

Conversion WebP avec Sharp

Tous les JPEG du dossier public/img/ ont été convertis en WebP (format moderne, compression 10× plus efficace) avec redimensionnement automatique à 1400px de largeur max (couvre tous les cas d'usage).

Process : convertisseur Node.js avec Sharp
Outils
npm install sharp — CLI Node pour traitement d'images. Utilise libVIPS (ultra-rapide) pour conversion + redimensionnement sans installer ImageMagick ou FFmpeg
Résultat
21 fichiers JPEG → 21 fichiers WebP. Fichiers JPG conservés comme fallback (non exposés). Fichiers WebP versionés dans Git et déployés dans client/dist/img/ après build

Optimisation des URLs d'images — optimizeImageUrl()

La fonction utilitaire optimizeImageUrl() dans client/src/lib/utils.js s'adapte à la source de l'image. Elle traite trois cas :

1. Supabase Storage

Pour les URLs contenant supabase.co/storage/v1/object/public/ : redirige vers l'endpoint /render/image/ pour bénéficier de la transformation d'images (ajuste la largeur, applique la compression et convertit en WebP à la volée).

2. Unsplash (images externes)

Pour les URLs images.unsplash.com : ajoute les paramètres imgix ?w=...&q=...&fm=webp&fit=crop pour la même optimisation côté Unsplash.

3. Images locales /img/

Pour les chemins relatifs /img/*.jpg : remplace automatiquement l'extension .jpg → .webp pour servir la version compressée. Exemple : /img/Dodge-Charger.jpg devient /img/Dodge-Charger.webp.

Schéma : peu importe la source, optimizeImageUrl() garantit que le navigateur reçoit la version la plus légère et moderne possible.

Au-dessus du fold (visible immédiatement)

loading="eager" dit au navigateur de télécharger l'image immédiatement. loading="lazy" l'ajourte au moment où elle approche du viewport. fetchPriority="high" la demande en priorité réseau par rapport aux autres ressources.

Fonts — Non blocking avec preconnect

Les polices Barlow et Barlow Condensed sont chargées depuis Google Fonts. Avant : @import url(...) dans le CSS — découvert après le parsing du CSS, bloquerait le rendu. Après : liens en <head> avec rel="preconnect" et rel="preload" — découverts immédiatement.

client/index.html — Head
<link rel="preconnect">
Établit la connexion TCP à fonts.googleapis.com en parallèle du parsing HTML. Gain : ~100-200ms avant la première requête font
<link rel="stylesheet">
Requête non bloquante pour la feuille de styles Google Fonts. Rendu de la page peut commencer avant que les fonts soit prêtes (FOUT — Flash Of Unstyled Text acceptable)

Serveur statique — Express.js

En production, Express sert maintenant les fichiers statiques du build React depuis client/dist/. Avant : seules les requêtes /api/* étaient gérées ; le frontend et les images n'étaient pas servis.

1
Build Vite génère client/dist/ avec tous les assets (JS, CSS, images WebP) prêts pour production
2
Express en production : app.use(express.static(distPath)) sert client/dist/ sur /
3
Les images WebP dans dist/img/ sont servies au même serveur que l'API (une seule connexion)
4
5
Fallback SPA : toute route non trouvée retourne index.html pour que React Router gère la navigation côté client

Résultats mesurés

Avant optimisations : 45 MB d'images pour 6 véhicules visibles → page 2-3 secondes à charger sur 4G
Après optimisations : 1.5 MB d'images pour les mêmes 6 véhicules → page < 500ms sur 4G

Les trois leviers de performance appliqués :

  1. Réduction du poids des fichiers (conversion WebP) — 95% du gain
  2. Reqêtes plus petites (dimensions adaptées par composant) — 5% du gain complémentaire
  3. Priorités intelligentes (eager/lazy + fetchPriority) — perception de rapidité immédiate

Architecture de production

1
Navigateur — envoie une requête HTTPS vers eclipse-auto.theo-ferrete.fr
2
Apache (port 443) — termine le SSL Let's Encrypt et sert les fichiers statiques React depuis client/dist

Pour les URLs /api/*, Apache fait un reverse proxy vers localhost:3001
4
Express (port 3001) géré par PM2 traite la requête et interroge Supabase
5
Supabase Cloud — base de données PostgreSQL hébergée externement
